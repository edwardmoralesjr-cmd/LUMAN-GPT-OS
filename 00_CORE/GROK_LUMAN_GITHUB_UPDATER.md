## 3. Core Rules

### 3.1 File & Directory Conventions

- Core meta-protocols and system-level documents live in `00_CORE/`
- Each GPT module keeps its own folder (e.g. `lucid-syntax-promo-pro/`, `roseborn-canon-guardian/`)
- Shared rules and voice live in `shared-core/`
- Use clear, lowercase-with-hyphens naming
- New major systems should follow the existing module template (instructions.md + deployment_instructions.md + changelog.md + test_prompts.md)

### 3.2 Commit Discipline

Every commit message must be:
- Clear and descriptive
- Prefixed with the affected area when helpful (e.g. `[COMMANDS]`, `[00_CORE]`, `[lucid-syntax-promo-pro]`)
- Include context for future self

Example:  
`[00_CORE] Add GROK_LUMAN_GITHUB_UPDATER.md protocol`

### 3.3 Safety & Security

**Never commit:**
- API keys, tokens, or credentials
- Physical or digital seed phrases
- Private financial data
- Employer-confidential information
- Any content that would cause harm if exposed

If sensitive material is accidentally requested, refuse and explain the boundary using LUMAN Voice.

### 3.4 Versioning & History

- Maintain clean changelogs inside each module
- Use this repository as the single source of truth for all custom GPT instructions
- When major updates occur, update `DEPLOYMENT_INDEX.md` and relevant module changelogs

## 4. Workflow When Creating or Updating Files

1. Read the current state of the target file or directory (when it exists)
2. Align new content with LUMAN Voice + Edward Style Rules
3. Use clean, well-structured Markdown with clear headings and copy-friendly sections
4. Create or update the file via the connected GitHub tool
5. Confirm the commit SHA and path back to Edward
6. Suggest next logical steps if relevant (e.g. update COMMANDS.md, add to project_index, create changelog entry)
